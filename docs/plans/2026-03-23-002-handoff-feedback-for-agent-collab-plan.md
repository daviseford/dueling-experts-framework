# Handoff to Claude: Gaps and contradictions to resolve in the hybrid handoff plan

## Why this handoff exists

The latest plan is strong overall, but it still contains a few material contradictions and unresolved design choices that are likely to cause implementation churn. Please produce a revised plan that resolves these items explicitly and removes any superseded details.

## Highest-priority issues

### 1) The plan still contains incompatible API/UI decisions
The document says the frontend reliability fix is to merge status into `/api/turns` and avoid split-brain, but earlier sections still define a separate `/api/status` endpoint. It also says `?after=<turn-id>` is supported, then later defers that optimization for v1. Likewise, interjection handling is described in three incompatible ways:
- max 5 queued with HTTP 429
- queue position responses (`{ queued: true, position: 2 }`)
- simplified v1 response `{ ok: true }` with no max size / no 429

Please choose one v1 contract and scrub the rest of the document so there is only one source of truth.

### 2) `session.md` mutability is acknowledged but not actually resolved
The plan calls out that `session.md` mixes immutable config with mutable `session_status`, but the implementation still depends on mutating `session.md` during runtime and recovery. Please decide one of these and update all phases consistently:
- keep `session.md` mutable and define the mutation contract clearly, or
- split runtime state into a separate file such as `state.json`

Right now the plan documents the problem but leaves the implementation half on each side.

### 3) Concurrent sessions are still under-specified
The plan explicitly notes that concurrent sessions conflict on port 3334, but it never decides whether v1 supports:
- exactly one session per repo,
- exactly one watcher server per process,
- dynamic port selection,
- lockfiles to prevent duplicate runs in the same repo,
- recovery behavior when another live process already owns the session.

Please make concurrency rules explicit. Without that, session startup, recovery, and UI discovery are all ambiguous.

### 4) “Human override” exists in requirements but has no protocol
R11 says the same agent resumes after escalation “unless human overrides,” but there is no defined way for the human to override anything. The POST schema only accepts `{ content }`, and the UI only shows a text box. Please either:
- remove “unless human overrides” from v1, or
- define an explicit override mechanism (API field, UI control, and orchestrator behavior).

### 5) Phase 1 and Phase 2 boundaries are not fully aligned
The intro says the phases were merged to “build UI with interjection from start,” but the actual implementation phases still keep watcher UI and interjection in Phase 2. Please make the phase story consistent: either Phase 1 is headless core loop only, or UI/interjection are truly included from the beginning.

## Important design questions

### 6) The 100K truncation rule conflicts with “all prior turns”
The plan says each prompt includes all prior turns, then says the oldest turns are truncated once the assembled prompt exceeds 100K chars. That is reasonable, but it is no longer “all prior turns.” Please redefine the requirement more precisely, for example:
- include full transcript while under the cap,
- otherwise include session brief + truncation notice + retained suffix window.

Also specify whether truncation can split a turn body or must drop whole turns.

### 7) Final artifact generation needs failure handling
`final-summary.md` is generated by invoking Claude again at session end, but the plan does not define timeout, retry, or fallback behavior for that final synthesis step. If Claude hangs or outputs invalid content there, session completion becomes ambiguous. Please define whether artifact generation is best-effort, retried, skippable, or able to fail the session.

### 8) Crash recovery says “offer to resume,” but the CLI is otherwise non-interactive
Please specify how recovery works operationally:
- prompt on stdin interactively,
- require a CLI flag such as `--resume <session-id>`,
- auto-resume the only recoverable session,
- or print resumable sessions and exit.

The current wording is too vague for implementation.

### 9) Windows compatibility needs a pass
The plan claims Windows 11 + Node 18 support, but some security/file-system details appear Unix-centric or underspecified:
- `0600` file permission language does not map cleanly to Windows
- `crypto.randomUUID()` commentary references Node 19/20 even though Node 18 is the stated floor
- atomic rename behavior is mentioned as a risk but not operationally constrained

Please normalize the implementation notes to what is actually expected on Windows.

### 10) Claude invocation should be settled before coding
The plan still contains multiple Claude invocation patterns (`--print -p`, stdin, older variants in narrative). Please decide the one supported v1 invocation, plus the fallback if that command shape fails on a user machine.

## Lower-priority cleanup items

### 11) Security hardening should distinguish required v1 work from aspirational notes
The plan mixes firm implementation requirements with suggestions like sanitizing `--topic`, escaping `---` in interjections, strict Origin checks on every request, and file permission enforcement. Please separate:
- must-build v1 controls,
- nice-to-have hardening,
- platform-dependent items.

### 12) The state-machine section and the “simple while loop” section still pull in different directions
The document says the 7-state machine is mostly conceptual and recommends a simpler loop for v1, but still presents the richer state machine as if it were normative. Please choose one canonical mental model for the revised plan.

## Suggested output for the next planning pass

Please return a revised implementation plan that:
1. removes superseded alternatives,
2. defines one canonical API contract for watcher/interjection behavior,
3. resolves session state storage and recovery semantics,
4. states concurrency/port/lock behavior explicitly,
5. either defines or removes human override behavior,
6. clarifies truncation and artifact-failure semantics,
7. aligns all phase descriptions with the actual scope.

## My recommendation

If you want the cleanest v1, I would recommend:
- single orchestrator process per session
- one watcher server per process
- dynamic port with printed URL
- lockfile per session or repo
- no human override in v1
- no `/api/status`
- no `?after` in v1
- one simple `{ ok: true }` interjection response
- mutable `state.json` separate from immutable `session.md`
- whole-turn truncation only
- best-effort `final-summary.md` generation with timeout + visible warning artifact on failure
